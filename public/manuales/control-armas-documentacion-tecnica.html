1. Alcance y principios

El módulo sincroniza funcionarios, armas y chalecos desde la base externa personal911, pero registra las retenciones y su auditoría en la base principal de la aplicación.

Fuente externa
Se consulta en modo lógico de lectura. El importador no ejecuta altas, cambios ni bajas sobre personal911.
Persistencia local
Los datos se normalizan en inventarios y asignaciones con historial.
Idempotencia
Ejecutar varias veces la sincronización no genera asignaciones activas duplicadas.
Historia inmutable
Cada retención guarda una instantánea del arma y del chaleco al momento del alta.

3. Conexión externa

La conexión se declara como personal911 en config/database.php. Las credenciales se proporcionan mediante variables de entorno:

PERSONAL911_DB_HOST=127.0.0.1
PERSONAL911_DB_PORT=3306
PERSONAL911_DB_DATABASE=personal911
PERSONAL911_DB_USERNAME=usuario
PERSONAL911_DB_PASSWORD=contraseña
No incluir credenciales reales en Git. Deben existir solamente en .env. Para producción se recomienda un usuario MySQL con privilegios exclusivamente SELECT sobre personal911.

4. Flujo de importación

Selección de origen

Se consultan funcionarios con Id_Estado = 2 (Activo Efectivo). La consulta incorpora el tipo de arma y la jerarquía mediante uniones a tipo_armas y jerarquias.

Identificación

Primero se busca por personal911_id, que es el identificador estable externo. En la primera vinculación puede utilizarse el L.P.; una colisión entre identificadores detiene la transacción.

Reconciliación

  1. Actualiza nombre, apellido, L.P. y Nom_Jerarquia.
  2. Crea o actualiza el arma por número único.
  3. Cierra asignaciones anteriores cuando cambia el funcionario o el bien.
  4. Crea la asignación activa si no existe.
  5. Extrae y asigna el chaleco cuando Obs_Func contiene información reconocible.
  6. Cierra asignaciones de bienes que ya no aparecen.
  7. Elimina lógicamente al personal que deja de estar en el conjunto activo.

Duplicados

Si dos funcionarios activos informan el mismo número de arma, el arma se registra pero no se asigna. El conflicto aparece en la salida del comando. Así se evita elegir un propietario arbitrariamente.

Un conflicto debe corregirse en la fuente personal911. La siguiente sincronización resolverá automáticamente la asignación.

Correcciones locales protegidas

Cuando un armero edita arma o chaleco de un funcionario importado, personals marca el recurso como bloqueado para importación. Mientras el dato de personal911 sea distinto, el importador no pisa el valor local y crea o actualiza una fila activa en inventario_discrepancias.

Si en una sincronización posterior Personal 911 coincide con el valor local, la discrepancia se marca como resuelta y se libera el bloqueo correspondiente.

5. Extracción de chalecos

La fuente no posee columnas estructuradas para chalecos. El parser inspecciona funcionarios.Obs_Func y reconoce variantes de texto para obtener:

seriemarcamodelotallenivellote

Chaleco Antibala ABPC Nro Serie: 16515 Talle: M
CHALECO BALISTICO SEATLE FORCE 10 SERIE N° 6668
CHALECO: N° 359 TALLE S-RB3 LOTE 24

Si el texto menciona un chaleco pero no permite identificar su serie, el registro se informa como observación pendiente y no se crea un bien incompleto.

Los índices únicos compuestos sobre (personal_id, activa) y (arma_id, activa) impiden más de una asignación activa por funcionario o arma. Las filas históricas utilizan activa = NULL.

7. Creación de una retención

  1. El controlador lista personal con arma activa y sin otra retención abierta.
  2. La petición valida funcionario, motivo, fecha y observaciones.
  3. El servicio obtiene arma y chaleco actuales.
  4. El motivo fija el tipo: RETENCIÓN, REGULACIÓN o RESGUARDO.
  5. Se crea el registro en estado EN_ARMERIA.
  6. Se calculan días restantes cuando corresponde.
  7. Se guardan referencias y snapshots: arma_numero, arma_tipo, chaleco_numero y chaleco_detalle.
  8. Se registra la acción y el usuario en auditoría.
Los snapshots son intencionales: una retención histórica no cambia visualmente cuando el funcionario recibe posteriormente otra arma o chaleco.

10. Operación y despliegue

Sincronización manual

php artisan armas:importar-personal911

Sincronización programada

Se agenda diariamente a las 05:30, con bloqueo contra ejecuciones superpuestas. El servidor debe ejecutar el scheduler de Laravel.

* * * * * php /ruta/artisan schedule:run >> /dev/null 2>&1

Salida y log

El comando informa funcionarios procesados, armas, chalecos, conflictos y observaciones no interpretadas. La ejecución programada escribe en storage/logs/armas_personal911.log.

Monitoreo en el dashboard

El endpoint api.dashboard.workers-status incluye conflictos activos y discrepancias activas para usuarios con ver-menu-armamento. La pestaña Novedades actualiza cada minuto los contadores y ofrece el detalle en modales separados.
